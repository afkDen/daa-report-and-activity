The Hidden Deadlock

Correct Answers
Task A — Visited Set:{ A, B, D, E }
Task A — Rec. Stack:[ A, B, D, E ]
Task A — Key Detail: C is NOT visited. DFS went A→B→D→E, never branching to C.
Task B — Topo Sequence:A → C → B → D → E
Task A detail: DFS starts at A, picks B first (alphabetical order, B before C). B→D→E is traversed in one deep chain. At E, the algorithm checks E's neighbor B — B is currently in the active recursion stack. Error fired immediately. C is untouched.

Task B (no E→B): dfs(A)→dfs(B)→dfs(D)→dfs(E): E has no neighbors → push(E). push(D). push(B). Back to A, next neighbor C → dfs(C): D is already visited → push(C). push(A). Stack: [E,D,B,C,A]. Top→bottom: A → C → B → D → E.
Task C — Formal Proof: No Valid Ordering Exists
①Edge B → D requires: B appears before D in any valid ordering.
②Edge D → E requires: D appears before E.
③Edge E → B requires: E appears before B.
④Chaining ①+②+③: B < D < E < B, meaning B must appear before itself.
✗ CONTRADICTION — No element in a finite linear sequence can precede itself. The cycle B→D→E→B makes a valid topological ordering mathematically impossible.

Correct Answers

Task A — Topo Sequence:S → B → A → T → Y → X
Task B — Push Order:[ Y, X, T, B, A, S ]

Task C — Total Orderings:4 distinct valid orderings
Task A trace (alphabetical — A before B, X before Y): dfs(S) visits A first. dfs(A)→dfs(T)→dfs(X): X has no children → push(X). dfs(Y): Y has no children → push(Y). T done → push(T). A done → push(A). Back to S: neighbor B → dfs(B): T already visited → push(B). S done → push(S). Stack [X,Y,T,A,B,S], top→bottom: S→B→A→T→Y→X.

Task B trace (reverse — B before A, Y before X): dfs(S) visits B first. dfs(B)→dfs(T)→dfs(Y): push(Y). dfs(X): push(X). push(T). push(B). Back to S: A → dfs(A): T visited → push(A). push(S). Stack [Y,X,T,B,A,S], top→bottom: S→A→B→T→X→Y.

Task C — Why 4? S has no incoming edges and must come first (only source). A and B have no ordering constraint between them — they can swap. X and Y have no constraint — they can swap. 2 × 2 = 4 total valid orderings. Note: Tasks A and B only produce Orderings 3 and 1 respectively; Orderings 2 and 4 are also valid but would require a mixed adjacency list order not used in either task.

04

The Stack Overflow Threat

Correct Answers
Task A — Chain Peak:6 frames
Task A — Call Chain:dfs(A)→dfs(B)→dfs(C)→dfs(D)→dfs(E)→dfs(F)
Task B — Star Peak:2 frames
Task B — Call Chain:dfs(A)→dfs(B) [then A→C, A→D, etc. — one leaf at a time]
Task C.i — Frames:10,000
Task C.ii — Memory:10,000 KB ≈ 9.77 MB
Task C.iii — Overflow?YES — 9.77 MB > 8 MB limit
Task D — Star 10K Peak:2 frames (constant, regardless of N)
Task A: The chain A→B→C→D→E→F forces DFS to descend the entire chain before any node returns. All 6 frames are simultaneously on the call stack when dfs(F) executes — this is the exact worst case for 6 nodes.

Task B: A star root (A) calls each leaf in sequence. dfs(B) returns before dfs(C) is called. At most 2 frames are ever active: dfs(A) and one leaf. Processing 999 leaves does not increase this maximum.

Task C calculation: 10,000 nodes in a chain → 10,000 simultaneous frames. 10,000 × 1 KB = 10,000 KB = 10,240,000 bytes = 9.765625 MB ≈ 9.77 MB. This exceeds the 8 MB stack limit. The runtime throws StackOverflowError before the algorithm finishes. This is why production-grade systems like npm use iterative topological sort (Kahn's algorithm), not recursive DFS.

Task D: A star's recursion depth is always 2 regardless of how many leaves exist. dfs(root) calls dfs(leaf), leaf returns immediately, root calls the next leaf. The call stack never grows beyond 2 frames. 9,999 leaves = still depth 2.

Key Insight: The worst-case recursion depth equals the length of the longest directed path in the graph — not the total vertex count. This is why real package managers like npm use Kahn's iterative (BFS-based) algorithm instead of recursive DFS: it avoids call-stack overflow entirely because it never uses recursion.

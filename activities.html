The Hidden Deadlock

CYCLE DETECTION · RECURSION STACK · WHAT-IF ANALYSIS

A CI/CD pipeline attempts topological sort on five microservices. The algorithm crashes with "CyclicDependencyError". Unlike a simple chain cycle, this graph has a fork-then-merge structure, meaning not all nodes are visited by the time the cycle is detected.

Pipeline Dependency Graph — Hidden Cycle: B → D → E → B
Edges: A→B, A→C, B→D, C→D, D→E, E→B. DFS starts from node A. Recall: only nodes in the active recursion stack trigger a cycle error — nodes that are merely in the visited set do not.
Task A: At the exact moment the back edge (E → B) is evaluated and the error is thrown, list the exact contents of both tracking structures. Note: not all nodes are visited yet when this occurs.

1. The permanent Visited Set (all nodes marked visited so far).
2. The temporary Active Recursion Stack (frames currently on the call stack).

Task B (What-If Analysis): The back edge E→B is removed. Execute DFS on the resulting acyclic graph (alphabetical neighbor order). Provide the exact topological sequence.

Task C (Proof): Without running the algorithm, prove that the original graph (with E→B) has no valid topological ordering. Show the specific dependency contradiction that makes it impossible.

The Double Diamond

MULTIPLE VALID ORDERINGS · ADJACENCY LIST MECHANICS

An assembly pipeline has one source (S), two parallel intermediate stages (A and B), a shared synchronization point (T), and two independent final outputs (X and Y). This structure — a "double diamond" — produces multiple equally valid orderings depending on how neighbors are iterated.

Pipeline Graph — S → {A,B} → T → {X,Y}
Edges: S→A, S→B, A→T, B→T, T→X, T→Y. The outer loop starts at S (the only unvisited source). T requires both A and B to be complete before it can proceed.
Task A (Standard DFS): Execute DFS with alphabetical neighbor ordering (S visits A before B; T visits X before Y). Provide the exact push order and the final topological sequence.

Task B (Reverse DFS): Now execute DFS with reverse-alphabetical neighbor ordering (S visits B before A; T visits Y before X). Provide the exact push order and the final topological sequence.

Task C (Count Orderings): How many distinct, mathematically valid topological orderings exist for this graph in total? List all of them.

The Stack Overflow Threat

WORST-CASE SPACE COMPLEXITY · RECURSION DEPTH · O(V)

The algorithm's worst-case space complexity is O(V) for the active recursion stack. But not all graphs with V vertices reach this depth. The two graph structures below have the same number of nodes but produce dramatically different recursion stack depths — with real-world consequences.

Graph A — Deep Chain (Worst Case)

A → B → C → D → … → N

depth grows with N
Graph B — Wide Star (Best Depth Case)

A → {B, C, D, E, F, …}

depth stays constant
Task A: For Graph A (chain of 6 nodes A→B→C→D→E→F), what is the exact maximum number of active call frames in the recursion stack at peak depth? Which call chain achieves this?

Task B: For Graph B (star: A→{B,C,D,E,F}), what is the exact maximum recursion stack depth? Which call chain achieves this?

Task C (Production Scenario): A package manager runs DFS-based topological sort on 10,000 packages with a single linear chain of dependencies (structured like Graph A). Assume each recursive call frame occupies 1 KB of stack memory. The runtime's call stack limit is 8 MB.

   i. How many frames are active in the recursion stack at peak depth?
   ii. How much total stack memory does this consume?
   iii. Does this exceed the 8 MB limit? Does a stack overflow occur?

Task D: If those same 10,000 packages were restructured as a star (Graph B pattern — one root with 9,999 direct leaves), what would the peak recursion depth be? Why is this structurally safe?
